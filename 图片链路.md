**整体思路**

可以把这条图片上传链路想成一个“外卖预点单”系统。

- 前端负责“下单”
- Java 后端是“前台接单员”
- OCR 服务是“后厨先做预处理”
- Agent 是“主诊医生”

关键优化点在于：  
**用户一上传图片，Java 就先偷偷让 OCR 开始干活。**  
这样等 Agent 真正思考到“我要看图”这一步时，OCR 结果大概率已经在 Redis 里热好了，不用再现场等识别。

---

**完整链路，按时间顺序讲**

**第一步：用户上传图片**

前端在 [ChatWindow.vue](/d:/Users/xiaoli/Desktop/MedLabAgent/frontend-vue/src/components/ChatWindow.vue) 点“上传”，通过 [ApiService.js](/d:/Users/xiaoli/Desktop/MedLabAgent/frontend-vue/src/services/ApiService.js) 发起：

```text
POST /api/v1/agent/upload-report
```

这里传的是 `multipart/form-data`，也就是浏览器真正把图片文件发给 Java。

这一步的角色可以理解为：

- 用户把化验单交到前台
- 前台先收件，不马上给最终分析

---

**第二步：Java 接住文件并保存**

Java 后端的 [AgentController.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/controller/AgentController.java) 收到上传后，把事情交给 [MedicalFacadeService.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/service/MedicalFacadeService.java)。

`MedicalFacadeService` 又调用 [StorageService.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/service/StorageService.java)，把图片先存进本地上传目录。

这时 Java 做了两件很关键的事：

1. 生成给浏览器看的地址  
   例如：

   ```text
   http://localhost:8080/api/v1/file/view/xxx.png
   ```

2. 生成给 OCR 容器看的地址  
   例如：
   ```text
   http://host.docker.internal:8080/api/v1/file/view/xxx.png
   ```

为什么要分两个地址？

因为：

- 浏览器访问 Java，当然走 `localhost:8080`
- 但 OCR 在 Docker 容器里，它眼里的 `localhost` 是它自己，不是宿主机 Java
- 所以它必须通过 `host.docker.internal:8080` 回到宿主机拿图片

这一步就像：

- 前台接单后，不但给顾客一张取号单
- 还给后厨一张内部取货单，告诉它去哪拿原始食材

---

**第三步：Java 立即异步预热 OCR**

这一步是整套设计最“聪明”的地方。

在 [MedicalFacadeService.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/service/MedicalFacadeService.java) 里，文件刚保存完，Java 就立刻调用 [VisionService.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/service/VisionService.java)。

`VisionService` 用异步方式触发 [OcrServiceClient.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/service/OcrServiceClient.java)，向 8001 发请求：

```text
POST /api/v1/analyze-vision
```

请求里带的是刚才那个 OCR 可访问 URL。

注意，这一步是**异步**的，意思是：

- Java 不会傻等 OCR 识别完再回前端
- 前端上传成功很快就能返回
- 但 OCR 已经在后台偷偷开工了

这就像：

- 前台一边跟顾客说“收到，请稍等”
- 一边已经把单子传进后厨
- 后厨开始切菜备料了

---

**第四步：OCR 服务去抓图、识别、写缓存**

OCR 服务在 [main.py](/d:/Users/xiaoli/Desktop/MedLabAgent/ai-services-python/ocr_service/main.py)。

它收到 Java 的 `path` 后，会先做一件事：  
**查 Redis 有没有现成结果。**

如果 Redis 已经有：

- 直接返回缓存
- 几乎瞬间完成

如果 Redis 没有：

1. OCR 服务用 HTTP 去访问：
   ```text
   http://host.docker.internal:8080/api/v1/file/view/xxx.png
   ```
2. 把图片字节下载下来
3. 转成 base64
4. 调 DashScope 视觉模型识别
5. 解析成结构化结果
6. 再把结果写入 Redis

所以 Redis 在这里扮演的是：

- “OCR 热缓存仓库”
- 相同图片第二次分析时，不用再重复识别

这一步就像：

- 后厨收到单子后先看“这道菜是不是刚做过”
- 如果做过，直接拿现成备菜结果
- 如果没做过，就重新切菜、配菜、装盘
- 然后把处理好的半成品放进保温柜备用

---

**第五步：前端把图片地址发给 Agent**

上传成功后，前端不是直接等 OCR 返回，而是把上传接口返回的 `filePath` 自动拼成一句自然语言：

```text
请分析这张化验单：http://host.docker.internal:8080/api/v1/file/view/xxx.png
```

这句会发给 8000 的 Agent。

也就是说，用户视角看到的是：

- 我上传完图片
- 系统立刻把这张图当作问诊输入的一部分交给 Agent

这一步就像：

- 顾客不仅把食材交给前台
- 还顺手说了一句：“请根据这份化验单告诉我结果”

---

**第六步：Agent 思考到要看图时，再调用 OCR**

Agent 在 [tools.py](/d:/Users/xiaoli/Desktop/MedLabAgent/langchain_service/tools.py) 里有 `AnalyzeMedicalImage` 工具。这个工具底层走 [vision_analyzer.py](/d:/Users/xiaoli/Desktop/MedLabAgent/langchain_service/vision_analyzer.py)。

当大模型推理到“这题必须看图”时，会调用 8001：

```text
POST /api/v1/analyze-vision
```

这次带的还是同一张图的 URL。

但和第一次不同的是：

- 很大概率 Java 在上传时已经提前触发过 OCR 了
- 所以 Redis 里已经有结果
- 8001 直接命中缓存秒回

于是用户感觉会像：

- Agent 好像一想到“我要看图”
- 图像结果就立刻出来了

其实不是模型超神，而是后端提前把 OCR 做完了。

这一步就像：

- 医生刚说“把这张片子拿来我看看”
- 护士已经提前把报告放桌上了

**遇到的主要困难，以及为什么难**

**1. 改错了上传入口**
一开始以为前端上传走的是 `FileController`，结果实际前端走的是：

```text
ApiService -> /api/v1/agent/upload-report -> AgentController -> MedicalFacadeService
```

所以前面明明改了 `FileController`，前端拿到的地址还是旧的。

为什么难？
因为系统里有两个“看起来都像上传入口”的地方：

- 一个是 `/api/v1/file/upload-report`
- 一个是 `/api/v1/agent/upload-report`

最后真正生效的是后者。

解决方法：

- 顺着前端调用链追到 [ApiService.js](/d:/Users/xiaoli/Desktop/MedLabAgent/frontend-vue/src/services/ApiService.js)
- 再追到 [AgentController.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/controller/AgentController.java)
- 最终改到 [MedicalFacadeService.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/service/MedicalFacadeService.java)

---

**2. Java 和 OCR 不在同一个文件世界**
Java 是本机跑的，OCR 是 Docker 容器跑的。

最开始传的是本地路径，比如：

```text
D:\Users\xiaoli\Desktop\MedLabAgent\uploads\xxx.png
```

对 Java 来说这个路径没问题，  
但对 Docker 容器里的 OCR 来说，这个路径根本不存在。

为什么难？
因为“同一台电脑”不等于“同一个文件系统视角”。

- 宿主机的 `D:\...`
- 容器里的 `/app/...`
  不是一回事

解决方法：

- 不再让 OCR 读宿主机本地路径
- 改成走 HTTP 取图
- Java 返回 `OCR_ACCESS_URL + /api/v1/file/view/{fileName}`

这样 OCR 只需要会“访问 URL”，不需要知道宿主机磁盘长什么样。

---

**3. `localhost` 在容器里是错的**
浏览器里 `localhost:8080` 指的是你电脑上的 Java。  
但在 OCR 容器里，`localhost:8080` 指的是容器自己。

所以如果 Java 把图片地址返回成：

```text
http://localhost:8080/api/v1/file/view/xxx.png
```

OCR 容器去访问时就会失败。

解决方法：

- 专门区分两个地址
- 给浏览器：`SERVER_URL=http://localhost:8080`
- 给 OCR：`OCR_ACCESS_URL=http://host.docker.internal:8080`

这个改动非常关键，它相当于：

- 给顾客一张外部导航图
- 给内部配送员一张员工通道图

---

**4. Java 文件接口被安全策略拦了**
后来 OCR 虽然能访问正确 URL 了，但还是抓不到图。

根因是 Java 的 Spring Security 没有放行：

```text
/api/v1/file/view/**
```

所以 OCR 容器请求图片时，被鉴权挡在门外。

解决方法：
在 [SecurityConfig.java](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/java/com/medlab/config/SecurityConfig.java) 放行：

- `/api/v1/file/view/**`
- `/api/v1/file/health`

这一步的本质是：

- 文件预览接口既然要给 OCR 容器匿名访问
- 就不能再要求登录态

---

**5. 本机代理把 localhost 请求搞乱了**
你机器上有代理环境变量，导致 LangChain 请求 `http://localhost:8001` 时，实际上先经过了代理，出现过 502/404 这种非常迷惑的现象。

为什么难？
因为表面看是“8001 挂了”，实际上是请求压根没按原路直连。

解决方法：
在 [vision_analyzer.py](/d:/Users/xiaoli/Desktop/MedLabAgent/langchain_service/vision_analyzer.py) 里用：

```python
trust_env=False
```

这样请求不会再自动继承系统代理环境变量。

---

**6. 上传大图时被 413 拦住**
图片开始能传，但一传大一点的文件就出现：

```text
413 Payload Too Large
```

这说明请求在进入业务代码前，就被 Spring multipart 默认限制挡掉了。

解决方法：
在 [application.yml](/d:/Users/xiaoli/Desktop/MedLabAgent/backend-java/src/main/resources/application.yml) 显式加：

- `spring.servlet.multipart.max-file-size`
- `spring.servlet.multipart.max-request-size`

把上传大小上限提到 100MB。

---

**7. OCR Docker 构建太重**
一开始 OCR 依赖里有 `paddleocr`、`PyMuPDF` 一类重包，构建镜像时非常慢，还容易出错。

解决方法：
精简 [requirements.txt](/d:/Users/xiaoli/Desktop/MedLabAgent/ai-services-python/ocr_service/requirements.txt)，只保留当前架构真正需要的包：

- `fastapi`
- `uvicorn`
- `httpx`
- `redis`
- `python-dotenv`
- 多模态 API 所需依赖

这样 OCR 变成“远程视觉模型识别 + Redis 缓存服务”，而不是本地重型 OCR 引擎。

---

**最后这套方案为什么值钱**

因为它不只是“能上传图片并识别”。

它实际上做到了三件更像工程系统的事情：

1. **上传与识别解耦**  
   上传先成功返回，不阻塞用户等待 OCR。

2. **感知预取**  
   Java 在用户还没真正发起分析前，就已经让 OCR 开始干活。

3. **缓存命中提速**  
   Agent 真正需要图像结果时，Redis 大概率已经热好，响应体验几乎像“瞬间看懂图片”。

所以从系统设计角度看，这不是一条普通的“上传 -> 调模型”链路，  
而是一条：

**“上传即预热，分析时命中缓存，微服务通过 HTTP 解耦文件访问，最终让 Agent 看起来像实时拿到图像理解结果”** 的链路。
