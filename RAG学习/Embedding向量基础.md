## Embedding 向量基础与检索分类（超详细面试讲解）

在 RAG、相似性搜索、推荐系统等任务中，Embedding 和检索技术是核心支柱。本章从向量基础概念到工业级混合检索，全面讲解。

---

## 第一部分：Embedding 向量基础

### 一、什么是稠密向量（Dense Vector）

**稠密向量**：用一个固定长度、元素为实数的数组来表示文本、图像、音频等对象的语义。与稀疏向量（如 one-hot 或词袋模型）不同，稠密向量几乎每个维度都非零，维度通常在 128~4096 之间。

**示例**（二维语义空间，简化）：
- "国王" → [0.9, 0.8]  
- "王后" → [0.85, 0.75]  
- "苹果" → [-0.2, 0.6]

通过神经网络（如 BERT、text-embedding-3-small）将输入映射到向量空间，使得**语义相近的对象在向量空间中距离较近**。

**像 One-Hot、词袋这类稀疏向量，只有极少维度有效、其余全为 0，只记录表层特征；稠密向量压缩融合了深层语义、上下文、关联关系，表达能力更强，是大模型、检索、AI 特征提取的核心基础。**

#### 1.1 稠密向量的生成方式

| 模型/方法 | 特点 | 维度 | 适用场景 |
|-----------|------|------|----------|
| **Word2Vec / GloVe** | 静态词向量，无法一词多义 | 50-300 | 传统 NLP |
| **BERT / RoBERTa** | 动态上下文向量，取 [CLS] 或平均池化 | 768-1024 | 通用文本表示 |
| **SBERT (Sentence-BERT)** | 专门针对句子/段落相似度优化 | 384-768 | 语义搜索、聚类 |
| **OpenAI text-embedding-3** | 大型模型，多语言，可降维 | 1536（大）、3072（最大） | 商业化 RAG |
| **Cohere Embed** | 多语言，压缩 256 维 | 256-4096 | 企业搜索 |
| **BGE / M3E** | 中文优化 | 768-1024 | 中文 RAG |

#### 1.2 稠密向量的核心性质

- **语义相似性保持**：相似文本的向量余弦相似度高。
- **固定维度**：便于存储和计算（如向量数据库）。
- **可加性**：向量平均往往代表“混合语义”（如文档 + 查询的融合表示）。
- **可降维**：通过 PCA 或 Matryoshka 表示学习可截取前几维仍保持性能。

### 二、余弦相似度（Cosine Similarity）

#### 2.1 定义与公式

给定两个向量 \( A, B \in \mathbb{R}^n \)，余弦相似度定义为：

\[
\text{cosine\_sim}(A, B) = \frac{A \cdot B}{\|A\| \|B\|} = \frac{\sum_{i=1}^{n} A_i B_i}{\sqrt{\sum_{i=1}^{n} A_i^2} \cdot \sqrt{\sum_{i=1}^{n} B_i^2}}
\]

**范围**：[-1, 1]  
- 1 表示方向完全相同（平行）  
- 0 表示正交（无相关性）  
- -1 表示方向相反（但文本向量通常不会为负，因为输出经 ReLU 或余弦嵌入约束）

**几何意义**：两个向量夹角的余弦值，忽略长度，只关注方向。

#### 2.2 为什么余弦相似度常用于 Embedding？

| 优点 | 解释 |
|------|------|
| **不受向量长度影响** | 文本长度不同导致向量模长差异大，但方向反映语义，余弦相似度对此鲁棒。 |
| **范围固定，易解释** | 0~1 之间（正数），可直接用作分数。 |
| **计算高效** | 点积后除以模长，复杂度 O(n)；与内积效率相当。 |

**归一化后的等价关系**：如果两个向量都已**单位化（L2 归一化）**，则余弦相似度 = 点积。

#### 2.3 计算示例

向量 A = [1, 2, 3]，B = [2, 4, 6]（B = 2A）  
- 点积 = 1×2 + 2×4 + 3×6 = 2 + 8 + 18 = 28  
- \(\|A\| = \sqrt{1+4+9}=\sqrt{14} \approx 3.742\)，\(\|B\| = \sqrt{4+16+36}=\sqrt{56} \approx 7.483\)  
- 余弦相似度 = 28 / (3.742×7.483) = 28 / 28 = 1.0 ✅

向量 A = [1, 0, 0]，B = [0, 1, 0]  
- 点积 = 0 → 余弦相似度 = 0 (正交)

### 三、欧氏距离（Euclidean Distance）

#### 3.1 定义

\[
\text{Euclidean}(A, B) = \sqrt{\sum_{i=1}^{n} (A_i - B_i)^2}
\]

范围：[0, +∞)，值越小表示向量越接近。

#### 3.2 与余弦相似度的对比

| 维度 | 余弦相似度 | 欧氏距离 |
|------|------------|----------|
| **对长度的敏感度** | 不敏感（仅方向） | 敏感（包含长度差异） |
| **取值范围** | [-1, 1]（或归一化后 [0,1]） | [0, ∞) |
| **典型应用场景** | 文本相似度、推荐 | 聚类、异常检测、图像检索 |
| **归一化影响** | 归一化后等价于点积 | 归一化后等价于余弦距离的单调变换 |

**重要关系**：当向量满足 L2 归一化（\(\|A\|=\|B\|=1\)）时：
\[
\|A - B\|^2 = 2 - 2 \cos(A,B)
\]
即欧氏距离与余弦距离单调相关，此时排序等价。

#### 3.3 何时优先使用欧氏距离？

- **聚类（K-Means）**：默认使用欧氏距离，因为基于质心的更新公式依赖平方和。
- **需要区分向量模长**：例如，文档长度可能携带重要信息（长文档 vs 短摘要），但通常文本 Embedding 会做归一化。
- **异常检测**：离群点的欧氏距离通常显著大于正常点。

**工业经验**：对于归一化后的 Embedding，两者排序结果一致，性能也几乎无差别。因此很多向量数据库（如 Milvus、Pinecone）同时支持两种距离计算。

---

### 四、向量归一化（Normalization）

#### 4.1 为什么要归一化？

- **消除长度影响**：使相似度只由方向决定，符合语义比较需求。
- **加速计算**：归一化后只需计算点积（=余弦相似度），少一次除法。
- **改善检索质量**：实验表明，对 Embedding 做 L2 归一化能提升在欧氏距离和点积下的检索一致性。

#### 4.2 常用归一化方法

**L2 归一化（最常用）**：
\[
\hat{A} = \frac{A}{\|A\|_2}
\]
结果：所有向量的 L2 范数为 1。

**L1 归一化**（用于概率分布）：\( \hat{A}_i = \frac{|A_i|}{\sum_j |A_j|} \)，输出和为 1。

**标准化（Standardization）**：减去均值除以标准差，但 Embedding 通常不这么做（会破坏相对分布）。

#### 4.3 Embedding 归一化的实际做法

- **模型输出后处理**：从 BERT 得到向量 → `normalized_vec = vec / np.linalg.norm(vec)`。
- **在训练时约束**：对比学习（如 SimCSE）直接优化余弦相似度，隐式要求归一化。
- **向量数据库存储**：通常存储归一化后的向量，以点积距离快速匹配。

**面试点**：归一化会丢失长度信息，如果长度本身是特征（比如 TF-IDF 向量中的词频强度），则不应该归一化。但语义 Embedding 中长度往往无意义或负面影响相似度。

## 第二部分：检索分类

### 一、稀疏检索：BM25 与关键词检索

#### 1.1 基本原理

**稀疏向量**：维度等于词表大小（或 n-gram），大部分维度为 0。每个文档/查询表示为词频（TF）或 TF-IDF 权重的向量。检索时计算查询与文档的匹配分数。

**BM25（Okapi BM25）** 是稀疏检索的黄金标准，基于概率检索框架，公式：

\[
\text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t, d) \cdot (k_1 + 1)}{f(t, d) + k_1 \cdot (1 - b + b \cdot \frac{|d|}{\text{avgdl}})}
\]

其中：
- \( f(t, d) \) = 词 t 在文档 d 中的出现次数
- \( |d| \) = 文档长度（词数）
- avgdl = 平均文档长度
- \( k_1, b \) 为调节参数（通常 k1=1.2~2.0, b=0.75）

**IDF(t)** = \(\log \frac{N - n_t + 0.5}{n_t + 0.5}\)，N 为总文档数，\( n_t \) 为包含 t 的文档数。

---

#### 1.2 大白话 + 拆分公式 + 极简数值举例，一口气看懂 BM25

先记住核心结论：
**BM25 是稀疏检索（关键词检索）最强算法，替代老旧的 TF-IDF，是搜索引擎、ES 检索默认打分公式。**
输入：查询词 \( q \)、文档 \( d \)
输出：一个**相关度分数**，分数越高，文档越匹配搜索词。

##### 一、整体公式人话翻译

\[
\text{BM25}(q,d) = \sum_{t\in q} \text{IDF}(t) \cdot \frac{f(t,d)\cdot(k_1+1)}{f(t,d) \;+\; k_1\cdot\big(1-b+b\cdot\frac{|d|}{\text{avgdl}}\big)}
\]

逐块拆解：
1. \(\boldsymbol{\sum\limits_{t\in q}}\)：遍历**用户搜索里每一个关键词 t**，逐个算分、最后累加。
2. \(\boldsymbol{\text{IDF}(t)}\)：词的重要性 —— **冷门词权重高，烂大街通用词权重低**（和 TF-IDF 的 IDF 一样）。
3. \(\boldsymbol{f(t,d)}\)：词频，关键词 \( t \) 在当前文档 \( d \) 里**出现次数**。
4. \(\boldsymbol{k_1}\)：词频饱和系数，限制「词反复堆砌刷分」，词出现再多，分数不会无限暴涨。
5. \(\boldsymbol{b,\ |d|,\ \text{avgdl}}\)：
   - \( |d| \)：当前文档长度
   - \( \text{avgdl} \)：整个数据库**平均文档长度**
   - \( b \)：长度惩罚系数，控制「长文档天然占便宜」的问题

##### 二、BM25 解决了 TF-IDF 的两大硬伤

1. TF-IDF 漏洞1：一篇文章疯狂重复关键词（关键词堆砌），TF 无限变大，分数无限变高 → 垃圾内容排前面。  
   ✅ BM25：词频会**饱和压制**，堆词没用。

2. TF-IDF 漏洞2：长文档天生字数多、含词多，随便就比短文档分高，不公平。  
   ✅ BM25：加入**文档长度归一化**，过长文档自动扣分。

##### 三、每个模块单独通俗解释

1. **\(\text{IDF}(t)\) 逆文档频率**：稀有词更重要。  
   例：“的、是、我”到处都有 → IDF 很小，贡献分极低；“神经卷积、多智能体RAG”很少见 → IDF 很大，命中就高分。

2. **词频项 \(\dfrac{f(t,d)\cdot(k_1+1)}{f(t,d) + \dots}\)**：克制词频爆炸。词出现1次正常加分，出现100次分数只微微涨，不会翻倍刷分，防止恶意堆砌关键词作弊。

3. **文档长度项 \(\big(1-b+b\cdot\dfrac{|d|}{\text{avgdl}}\big)\)**：长短文档公平化。  
   - 文档 = 平均长度：系数接近 1，不扣分  
   - 文档**特别长**：系数变大 → 整体分母变大 → 分数**被压低**  
   - 文档**很短**：系数变小 → 轻微提分  

常用超参（背诵即可）：\( k_1 \approx 1.5,\quad b \approx 0.75 \)

##### 四、极简小例子（看懂计算逻辑）

设定：
- 查询 \( q \)：`人工智能`（关键词 \( t= \)人工智能）
- \( k_1=1.5,\ b=0.75 \)
- 平均文档长度 \( \text{avgdl}=1000 \)

1. 文档A（正常长度 1000字）：\( f(t,d)=3 \)，\(\text{IDF}=2.2\)
2. 文档B（超长水文 3000字）：\( f(t,d)=3 \)，\(\text{IDF}=2.2\)

- 文档A长度因子：\(1-0.75+0.75\cdot\dfrac{1000}{1000} = 1.0\)
- 文档B长度因子：\(1-0.75+0.75\cdot\dfrac{3000}{1000} = 2.5\)

仅看词频部分：
- 文档A该项：\(\approx 1.67\)
- 文档B该项：\(\approx 1.11\)

👉 **同样关键词出现次数，长文档直接被扣分**，非常合理。最后乘上 IDF、累加所有查询词，就是最终 BM25 总分。

##### 五、关键定位（结合前面的知识）

1. BM25 作用对象：**稀疏向量**（TF、词表、one-hot 类关键词向量）
2. 适用场景：传统检索、关键词匹配、ES/solr 搜索引擎
3. 对比：
   - BM25：**稀疏、关键词精准匹配**（找含这些词的文档）
   - 语义Embedding+向量检索：**稠密、语义匹配**（意思相近就算）

##### 六、一句话背诵版（面试直接用）

BM25 是稀疏关键词检索的经典打分模型，基于概率检索框架；结合**IDF词重要性、约束饱和词频、文档长度归一化**三大因子，解决了 TF-IDF 词频泛滥、长文档偏见的问题，是传统检索的工业界黄金标准。

#### 1.3 BM25 的优点与缺点

| 优点 | 缺点 |
|------|------|
| 无需训练，直接使用词频和逆文档频率 | 词汇鸿沟（同义词、多义词无法匹配） |
| 可解释性强（匹配具体关键词） | 对语序、语法不敏感 |
| 在小规模语料上速度快 | 无法理解语义相似但词不同的查询（如“汽车” vs “轿车”） |
| 支持精确匹配（法律、代码、专有名词） | 长查询可能会被词频放大噪声 |

#### 1.4 实现与工具

- **Elasticsearch**：内置 `match` 查询（based on BM25）
- **pyserini**：Python 接口，支持 BM25 及稀疏向量。
- **Whoosh**：轻量级索引库。

**面试示例**：查询“苹果发布新手机”，BM25 能匹配含“苹果”“手机”的文档，但无法匹配“iPhone 16”的文档（除非其中有“苹果”同现）。

### 二、稠密检索：Embedding 向量检索

#### 2.1 核心流程

1. **离线阶段**：所有文档 → 嵌入模型 → 归一化 → 存储到向量数据库。
2. **在线阶段**：查询 → 同一嵌入模型 → 归一化 → 在向量数据库中计算与所有文档向量的相似度（余弦/点积/欧氏） → 返回 Top-K。

**典型架构**：

```
文档库 → 分块 → Embedding模型 → 向量索引（HNSW/IVF）→ 存储
用户查询 → Embedding模型 → 查询向量 → 向量索引检索 → 相关文档块
```

#### 2.2 稠密检索的优点与缺点

| 优点 | 缺点 |
|------|------|
| 语义匹配能力强（同义词、多语言、上下文） | 需要训练或微调嵌入模型 |
| 对语序和句法有一定鲁棒性 | 计算开销大（索引构建、推理延迟） |
| 适合多模态（图文、音视频） | 可解释性差（黑盒） |
| 长尾查询效果优于稀疏检索 | 对专有名词、数字、代码不如 BM25 |

#### 2.3 典型模型与评测

**主流稠密检索模型**（用于 RAG）：
- **DPR（Dense Passage Retriever）**：基于 BERT 的双编码器，开山之作。
- **ColBERT**：晚期交互，精度高但速度慢。
- **Contriever**：无监督训练，简单好用。
- **BAAI/bge-large-en**：目前英文 SOTA。
- **BAAI/bge-large-zh**：中文 SOTA。
- **OpenAI text-embedding-ada-002**（1536维）及其后续版本 text-embedding-3。

**评测基准**：
- **BEIR**：19 个零样本检索任务。
- **MTEB**：涵盖 8 个任务类别，包括检索、聚类、重排序等。
- **C-MTEB**：中文版本。

#### 2.4 向量索引加速

暴力计算 KNN（线性扫描）在百万级以上数据时不可行。因此采用近似最近邻（ANN）算法：

| 算法 | 原理 | 特点 |
|------|------|------|
| **HNSW**（Hierarchical Navigable Small World） | 构建多层图，贪心搜索 | 召回率最高，构建慢，内存大 |
| **IVF**（Inverted File Index） | 聚类 + 倒排，只搜索少数聚类 | 速度快，内存低 |
| **IVF-PQ** | 乘积量化压缩向量 | 极低内存，适合十亿级 |
| **LSH**（Locality Sensitive Hashing） | 哈希桶 | 理论保证，工程少见 |

**工业选型**：几百万向量 → FAISS (IVF) 或 HNSWlib；亿级 → Milvus/Qdrant 使用 IVF-PQ 或 DiskANN。

---

#### 2.5 逐段大白话精讲（向量索引加速）

##### 核心前提

- **暴力KNN（线性扫描）**：用户查一个向量，要和库里**所有向量**逐个算距离、比相似度。数据量到**百万级、千万级**时，速度极慢、完全没法用，线上系统直接卡死。
- **解决方案：ANN 近似最近邻算法**：不遍历全部数据，只搜索「大概率相似」的向量，**牺牲一点点精度，换百倍速度**，是向量数据库、RAG 必用技术。

##### 四种索引算法通俗拆解

1. **HNSW 分层导航小世界图**  
   - 原理：给所有向量搭建**多层网络图**，搜索时从上到下贪心找邻近节点  
   - 特点：召回准确率最高、效果最好；但建索引慢、占用内存大  
   - 场景：RAG 最主流，几百万向量首选

2. **IVF 倒排文件索引**  
   - 原理：先用 KMeans 把全量向量聚成很多簇；检索时只搜**距离最近的几个簇**，不用扫全量  
   - 特点：速度快、内存占用低；FAISS 经典方案

3. **IVF-PQ**  
   - 原理：在 IVF 基础上，加**乘积量化**把向量压缩变小  
   - 特点：内存极低，专门应对**十亿级超大规模向量**

4. **LSH 局部敏感哈希**  
   - 原理：用哈希函数，把相似向量放进同一个哈希桶  
   - 特点：有理论数学保证，但工程落地少、效果一般，基本不用

##### 工业选型（背诵）

- 几百万向量：FAISS-IVF、HNSW
- 亿级海量数据：Milvus/Qdrant + IVF-PQ / DiskANN

### 三、混合检索（Hybrid Search）—— 工业标配

#### 3.1 为什么必须用混合检索？

两种检索各有致命短板，**互补才能完美**：

1. **BM25 稀疏检索**  
   ✅ 优点：字面精准匹配强，冷门专业词、数字、编号、专有名词、短语抓得极准  
   ❌ 缺点：**不懂语义**，同义词、近义词、改写句子完全找不到

2. **稠密向量检索（BGE/Embedding）**  
   ✅ 优点：懂语义、懂上下文，同义改写、意近文本完美匹配  
   ❌ 缺点：精准关键词、特殊编号、生僻词容易漏检，容易受输入错别字/噪声干扰

> 混合检索 = **稀疏BM25 + 稠密向量** 一起用，双路召回，取长补短。

#### 3.2 三种结果融合方法（把两份排名合并成一份）

##### 1）加权分数线性组合

\[
score(d)=\alpha \cdot norm(S_{BM25}) + (1-\alpha) \cdot norm(S_{dense})
\]

- 先把 BM25 分数、稠密向量分数做归一化（缩放到同一区间）
- \(\alpha\) 是权重（0.3~0.7），手动调参
- 简单直观，适合自定义调优场景

##### 2）RRF 倒数排名融合【业界最常用】

\[
RRF(d)=\sum \frac{1}{k+rank(d)},\quad k常取60
\]

- 不看具体分数，**只看文档排名位置**
- 排名越靠前，得分越高
- 最大优势：**不用归一化、不用调权重**，鲁棒性极强，ES、Milvus 原生支持

##### 3）LTR 学习排序

- 用 LambdaMART 等模型，输入 BM25分、稠密分、文档长度等特征
- 需要人工标注数据，适合大厂精细化优化，普通 RAG 项目不用

#### 3.3 标准实践流程（完整 RAG 链路，必背）

```
用户查询
├─ 稀疏支路 → BM25检索 → 取出 Top200 文档
└─ 稠密支路 → 向量检索 → 取出 Top200 文档
       ↓
两份结果合并 → RRF/加权融合排序
       ↓
重排模型Reranker精筛 → 选出最优Top10
       ↓
拼接上下文 → 输入大模型 → 生成答案
```

#### 3.4 工业现状

主流数据库全部原生支持**混合检索**：
- ES 8.x：自带 RRF 融合
- Milvus/Qdrant：原生稀疏+稠密混合搜索
- Weaviate、Pinecone：均支持稀疏向量 + 稠密向量联合检索

#### 3.7 混合检索的工业现状（同上已述）

**面试点**：混合检索通常能显著提升 Recall@10。据 Anthropic、LangChain 等实验，混合比单独使用 BM25 或稠密检索提升 5-20% 的命中率（MTEB 中平均提升 8%）。

- **资源限制极严**（移动端、边缘设备）：单一检索更简单。

### 五、面试常见问题与深度回答

#### Q1：什么是 Embedding？为什么它能用于检索？

> “Embedding 是将高维离散对象（如单词、句子）映射到低维连续向量空间的技术，使语义相近的对象在向量空间中距离更近。用于检索时，我们可以将文档和查询都编码成向量，然后通过计算向量相似度（如余弦相似度）来找到最匹配的文档。它解决了词袋模型无法处理的同义词、一词多义问题。”

#### Q2：余弦相似度与欧氏距离在 Embedding 检索中的区别？

> “余弦相似度只考虑向量的方向（角度），不受向量长度影响，适合比较不同长度的文本。欧氏距离同时考虑方向和长度。如果 Embedding 经过 L2 归一化（常见做法），两者单调相关，排序等价。在归一化前，如果文档长度本身是语义的一部分（如 TF-IDF 向量），则应使用欧氏距离；否则优先余弦相似度。”

#### Q3：BM25 为什么仍然是工业界标配？它有什么局限性？

> “BM25 无需训练，可解释性强，对精确关键词匹配（如产品型号、法律条款）效果远好于稠密检索。局限是无法理解语义，例如搜索‘智能手机’不会召回只包含‘iPhone’的文档，且对语序、否定词不敏感。”

#### Q4：混合检索如何融合 BM25 和稠密检索的结果？举例说明 RRF。

> “常用方法：加权和或倒数排名融合（RRF）。RRF 不依赖分数归一化，公式为 RRF(d) = Σ (1/(k+rank))，k=60。例如，一个文档在 BM25 中排第 2 名，在稠密中排第 5 名，则 RRF 分数 = 1/(60+2) + 1/(60+5) ≈ 0.0161 + 0.0154 = 0.0315。最后按 RRF 降序输出。这种方法稳定，能融合不同排序器的结果。”

#### Q5：在实际 RAG 系统中，通常如何设计检索部分？

> “工业 RAG 普遍采用混合检索 + 重排序两阶段：  
> 第一阶段：BM25 和稠密检索并行，各取 Top 200（或 1000），通过 RRF 融合得到候选集。  
> 第二阶段：使用更强大的重排序模型（如 Cohere rerank、BGE-reranker）对候选文档精细排序，取 Top 5 送入大模型。  
> 这样兼顾了召回效率和精度。”

#### Q6：向量数据库中的 HNSW 索引原理是什么？为什么比 IVF 快？

> “HNSW 构建多层图结构：底层包含所有节点，上层节点稀疏。查询时从最高层开始，贪心搜索最近节点，再下降到下层精细化。它的优势是 O(log N) 的平均复杂度，且图结构使得搜索路径短。相比 IVF（聚类后只扫几个桶），HNSW 在相同召回率下查询延迟更低。但构建速度慢，内存占用高。”

#### Q7：Embedding 模型如何选择？有中文场景的建议吗？

> “英文通用场景选 OpenAI text-embedding-3-small（性价比高）或 BAAI/bge-large-en；需要高精度可用 Cohere Embed v3。中文场景推荐 BAAI/bge-large-zh，或智源的 m3e-base。如果资源受限，可选用 text-embedding-3-small 降维到 256 维，或用 SBERT 蒸馏的小模型。离线部署选 bge-large-zh，API 选 OpenAI 或智谱 AI 的 embedding 接口。”

### 六、总结速记表（面试前复习）

| 概念 | 核心 |
|------|------|
| **稠密向量** | 固定长度实数向量，语义接近则距离近 |
| **余弦相似度** | 方向相似度，范围[-1,1]，归一化后等价于点积 |
| **欧氏距离** | 绝对距离，包含长度信息 |
| **L2 归一化** | 使向量模为1，消除长度影响 |
| **BM25** | 词频 + 逆文档频率 + 长度归一化 |
| **稠密检索** | 向量数据库 + ANN 索引（HNSW/IVF） |
| **混合检索** | BM25 + 稠密检索 + RRF/加权融合，工业标配 |
| **融合方法** | RRF（无参、鲁棒）或线性加权 |
| **评估指标** | Recall@K, NDCG, MRR |

### 极简总结（快速记忆）

1. 海量向量不能暴力检索，用 ANN 近似索引：HNSW 准、IVF 快、IVF-PQ 存海量；
2. 混合检索是 RAG 标配：BM25 管字面精准，稠密向量管语义理解；
3. 融合首选 RRF，无需归一化、效果稳；
4. 线上项目统一走「双路召回 + 融合排序 + 重排」标准流程。
